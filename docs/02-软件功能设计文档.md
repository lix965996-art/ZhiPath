## 1 引言

### 1.1 编写目的

本文档面向赛题评审、系统架构师、前后端开发与测试人员，目的是：

- 完整说明 ZhiPath 的总体架构与各层职责划分，使新成员可在不读全部源码的前提下建立全局认知。
- 固化接口协议、数据库表与 JSON 存储结构，作为前后端联调与数据迁移的权威依据。
- 把 BKT、DKT、IRT、FSRS、RAG 管线、知识图谱、画像抽取等核心算法的设计意图与实现要点讲透，支撑"基于大模型的精准评估"加分项的评审。
- 给出非功能设计（防幻觉、内容安全、多模型路由、凭据健康、全链路追踪、降级容错、代码沙箱隔离）的具体落地方式，证明工程深度。
- 沉淀部署与运行设计，使评委可在本地或容器环境中复现系统。

### 1.2 设计目标

ZhiPath 的设计围绕以下核心目标展开：

1. **精准评估**：用 BKT/DKT/IRT/FSRS 四套教育数据挖掘算法对学习者建模，回答"学生现在掌握了什么、下一步该学什么、什么时候该复习"。
2. **因材施教**：资源生成、测验出题、学习路径均以学习者画像与掌握度快照为输入，做到千人千面。
3. **多智能体协同**：用能力（Capability）+ Agent 编排框架，把目标诊断、资源生成、辩论、自主决策等复杂任务拆给多个专业 Agent 并行或接力完成。
4. **流式可观测**：所有交互走 WebSocket 流式协议，16 种事件类型把"思考—工具调用—中间结果—最终回答"全过程实时下发，前端可渲染工作流图与追踪时间线。
5. **安全可控**：本地正则护栏 + 代码沙箱双重防线，覆盖涉政暴恐、提示词注入、学术不端与学生 C 代码逃逸。
6. **优雅降级**：LLM 凭据缺失、Reranker 未安装、pgvector 不可用等场景下，系统自动降级而非崩溃。

### 1.3 术语

| 术语 | 含义 |
|---|---|
| Capability（能力） | 系统对一类用户意图的处理单元，如 chat、goal、resource_gen、auto_tutor 等，共 9 个 |
| Agent | 能力内部调用的专业生成/分析模块，如 QuizGenerator、GoalPlanner、Proposer 等 |
| UnifiedContext | 编排器入口数据传输对象，携带 session、消息、历史、画像、知识上下文等 |
| StreamBus | 单轮流式事件总线，能力向其 emit 事件，WS 层订阅并转发前端 |
| BKT | 贝叶斯知识追踪，4 参数（先验/学习/失误/猜测）的掌握度更新模型 |
| DKT | 深度知识追踪，基于 GRU 序列模型预测下一题答对概率 |
| IRT | 项目反应理论，2PL 模型，用于自适应题目难度选择 |
| FSRS | 自由间隔重复调度算法，17 参数，用于错题/闪卡的复习排程 |
| RAG | 检索增强生成，向量 + BM25 + 知识图谱混合检索 |
| KC | Knowledge Component，知识组件，BKT 追踪的最小掌握度单元 |
| MCP | Model Context Protocol，把 ZhiPath 能力暴露给外部 Agent 的标准协议 |
| xAPI | Experience API，学习经历记录标准（Statement = actor + verb + object + result + context） |
| pgvector | PostgreSQL 向量扩展，用于存储 768 维文档块嵌入 |

## 2 总体设计

### 2.1 体系架构

ZhiPath 采用前后端分离的三层架构，后端为单进程 FastAPI 应用，内部以"编排器—能力—Agent—服务"四级结构组织。整体分层用文字描述如下：

```
表现层（Next.js 浏览器端）
   │  WebSocket /api/v1/ws（流式交互）
   │  REST /api/v1/*（CRUD 与导出）
   ▼
应用层（FastAPI 单进程）
   ├─ 接入层：api/main.py 装配中间件、路由、能力注册、统一异常处理
   │   · CredentialsMiddleware：把 X-LF-* 请求头注入 contextvars
   │   · CORSMiddleware：放行 localhost:3000
   ├─ 编排层：runtime/orchestrator.py 的 ChatOrchestrator
   │   · 接收 UnifiedContext → 按 active_capability 查注册表 → 能力执行 → 流式事件回传
   ├─ 能力层：capabilities/*.py 的 9 个 Capability（chat/goal/learning/resource_gen/auto_tutor/agentic/debate/explainer/engineering_trace）
   ├─ Agent 层：modules/* 下的领域 Agent（资源生成 8 个、目标诊断 3 个、画像 1 个、路径 1 个、导师 1 个）
   └─ 服务层：services/* 下的领域服务（RAG、Mastery、SRS、KnowledgeGraph、CodeLab、Guardrail、Memory、Profile、xAPI、Tracing 等）
   ▼
数据层（双层持久化）
   ├─ PostgreSQL + pgvector：8 张关系/向量表
   └─ backend/data/ 下 18 个 JSON 目录：掌握度、DKT、知识图谱、复习卡片、资源包等
```

关键设计取舍：

- **单进程 + async**：FastAPI + asyncio 单进程即可承载赛题评审量级的并发；WebSocket 用原生 Starlette 实现，无需额外消息中间件。
- **能力注册表解耦**：编排器不硬编码任何业务逻辑，只做"查表—执行—转发"，新增能力只需 `registry.register()` 一行。
- **StreamBus 扇出**：能力产出的 `StreamEvent` 经 StreamBus 的 `asyncio.Queue` 扇出给订阅者，支持历史回放，保证后订阅的 WS 不丢事件。
- **双层持久化**：结构化强关系数据（用户、会话、消息、文档向量）入 PostgreSQL；高频读写的会话级状态（掌握度、复习卡片、知识图谱、画像）入 JSON 文件，避免锁竞争与迁移成本。

### 2.2 多智能体协同框架

ZhiPath 的多智能体协同体现在三个层次：

1. **能力间协同（Agentic 路由）**：`AgenticChatCapability` 让 LLM 自主决定何时把任务转交给 goal、learning、resource_gen、auto_tutor、debate 等能力，或查询学情（掌握度、待复习卡片、知识图谱）。工具循环上限 4 轮（`MAX_TOOL_ITERATIONS = 4`），每轮最多并行 4 个工具（`MAX_PARALLEL_TOOLS = 4`）。
2. **能力内多 Agent 并行**：`ResourceGenerationCapability` 用 `asyncio.gather` 并行调度 7 个生成 Agent（QuizGenerator、FlashcardGenerator、MindMapGenerator、CodeLabGenerator、MermaidGenerator、KGGenerator、CaseStudyGenerator），各自产出结构化资源后由编排者汇总。
3. **多角色辩论**：`DebateCapability` 用 Proposer（正方）、Skeptic（反方）、Judge（裁判）三角色进行 2 轮辩论，每轮双方各发言一次，最后由 Judge 流式输出综合裁定。

Agent 之间的结构化通信通过 `StreamBus.agent_message(from, to, payload, label)` 实现，前端 `AgentWorkflowGraph` 组件据此实时画连线气泡。能力产出则通过 `StreamEvent` 的 `RESULT`、`CONTENT`、`TOOL_CALL`、`TOOL_RESULT` 等事件类型回传。

### 2.3 模块划分

后端按职责划分为以下顶层包：

| 包 | 职责 |
|---|---|
| `api/` | FastAPI 应用入口、中间件、24 个路由文件 |
| `runtime/` | ChatOrchestrator 编排器、CapabilityRegistry 注册表 |
| `capabilities/` | 9 个能力实现 + BaseCapability 抽象基类 + PromptedLLMCapability 公共基类 |
| `core/` | UnifiedContext、EventType/StreamEvent、StreamBus 三大运行时核心数据结构 |
| `modules/` | 5 个领域子模块：chat_tutor、learner_profile、learning_path、resource_gen、skill_gap，各含 agents/ 与 prompts/ |
| `services/` | 领域服务：rag、mastery、srs、knowledge_graph、code_lab、guardrail、memory、profile、quiz、exam、resource_package、session、xapi、tracing、report、experiments、demo、auth |
| `config/` | schemas.py（AppConfig 等 dataclass）+ loader.py（读 default.yaml） |
| `base/` | 基础设施：llm_factory、model_router、credential_context、iflytek_factory、llm_retry、search_rag、base_agent |
| `rag/` | kb_loader、retriever（轻量检索入口） |
| `scripts/` | 数据导入脚本：ingest_408、ingest_course、ingest_pdf、verify_408、merge_duplicate_packages |

### 2.4 分层职责

- **接入层**负责协议适配（WS / REST）、凭据注入（contextvars）、统一错误响应（`error_type` 字段区分 http/validation/internal）、CORS。
- **编排层**负责单轮生命周期管理：创建 StreamBus、把异常翻译为 `CREDENTIAL_HEALTH` 与 `ERROR` 事件、WebSocket 断开时取消后台任务、保证最终一定 emit `DONE`。
- **能力层**每个 Capability 实现 `run(context, stream)`，内部决定调用哪些 Agent、走哪个模型路由、产出哪些事件。
- **Agent 层**是无状态函数式模块（`generate_*_with_llm`），接收 LLM 实例与参数，返回结构化 dict，不直接持有 StreamBus。
- **服务层**封装领域逻辑与持久化，对上提供 async 接口，对下操作 PostgreSQL 或 JSON 文件。

### 2.5 技术选型

| 层次 | 选型 | 理由 |
|---|---|---|
| 后端框架 | FastAPI + Uvicorn | 原生 async、WebSocket、自动 OpenAPI、生态成熟 |
| ORM | SQLAlchemy 2.x（异步） + asyncpg | 类型安全、支持 pgvector |
| 数据库 | PostgreSQL 16 + pgvector | 向量检索 + 关系数据一体 |
| LLM 编排 | LangChain（ChatModel 抽象 + bind_tools） | 多 provider 统一接口、原生 function calling |
| 嵌入模型 | sentence-transformers/all-mpnet-base-v2（768 维） | 开源、CPU 可跑、效果好 |
| Reranker | cross-encoder/ms-marco-MiniLM-L-6-v2 | 轻量、跨编码器、检索质量提升显著 |
| 前端框架 | Next.js 14（App Router）+ React 18 + TypeScript | SSR/CSR 混合、路由内建 |
| 3D 可视化 | Three.js / R3F | 知识星图、轨道 3D、数字人舞台 |
| 容器 | Docker Compose（postgres → backend → frontend） | 一键起服务、健康检查串联 |
| 其他 | Alembic（迁移）、pytest（测试）、MarkdownIt（文档转换） | 标准工具链 |

## 3 接口设计

### 3.1 通信总览

ZhiPath 同时提供两类接口：

- **WebSocket**（`/api/v1/ws`）：所有交互式对话、资源生成、辩论、Auto-Tutor 等流式能力的唯一入口。客户端发送 JSON 消息，服务端以 16 种事件类型的 JSON 消息流式回传。
- **REST API**（`/api/v1/*`）：用于 CRUD、查询学情、导出试卷/报告、上传文档、xAPI 记录、MCP 工具调用等非流式操作。共 24 个路由文件。

两类接口共享 `CredentialsMiddleware`：REST 通过 `X-LF-*` 请求头注入凭据，WebSocket 通过 `init`/`set_credentials` 消息注入，最终都落到同一组 contextvars，供 LLMFactory 与 ModelRouter 读取。

### 3.2 WebSocket 协议

#### 客户端 → 服务端消息类型

| type | 含义 | 关键字段 |
|---|---|---|
| `init` / `set_credentials` | 注入用户 LLM 配置 | `configs`（ApiConfig 列表）、`tts`（讯飞 TTS 凭据）、可选 `token` |
| `message` / `start_turn` | 发起一轮对话 | `content`、`session_id`、`capability` |
| `ping` | 心跳 | — |
| `cancel_turn` | 取消当前轮 | — |

服务端对 `init`/`set_credentials` 回 `credentials_ack`（含 config_count、tts_configured、authenticated）；对未知 type 回 `error`。

#### 服务端 → 客户端事件类型（16 种 EventType）

| 事件 | 用途 |
|---|---|
| `session` | 会话元信息（session_id） |
| `stage_start` / `stage_end` | 阶段开始/结束（能力内子流程边界） |
| `thinking` | 思考过程（前端灰色气泡展示） |
| `content` | 最终回答的流式文本块 |
| `tool_call` / `tool_result` | 工具/Agent 调用与返回 |
| `sources` | 检索来源（带 low_confidence 标记） |
| `result` | 结构化结果（如资源包摘要） |
| `error` | 错误信息 |
| `done` | 本轮结束（携带 trace_id） |
| `agent_message` | Agent 间结构化通信（from/to/payload/label） |
| `profile_update` | 画像维度增量更新 |
| `loop_step` | Auto-Tutor 闭环单步进度 |
| `credential_health` | API 凭据健康状态（rate_limited/auth_error/connection_error） |

#### StreamEvent 数据结构

`StreamEvent` 是 9 字段 dataclass：`type`、`content`、`source`、`stage`、`metadata`（dict）、`session_id`、`turn_id`、`seq`（自增序号）、`timestamp`。`to_dict()` 序列化为上述 WS JSON。

#### 单轮处理时序（`_handle_turn`）

WS 层在收到 `message` 后，按固定顺序组装 `UnifiedContext`：

1. 会话获取/创建（若无 session_id 则用消息前 30 字为标题建会话）。
2. 用户消息落库（`session_store.add_message`）。
3. 输入安全检查（`check_content_safety`）；命中 block 则下发 `guardrail` 事件并终止。
4. 画像更新（`profile_service.update_from_user_message`）并下发 `profile_update` 增量。
5. 记忆上下文组装（画像上下文 + 长期记忆摘要）。
6. SmartRetriever 检索知识上下文（带引用编号），命中则下发 `sources`。
7. 对话历史构建（排除当前句）。
8. 组装 `UnifiedContext`，下发 `session` 事件。
9. 在 `trace_scope` 内运行编排器，流式转发所有事件，累积 `content` 作为助手回复落库。

## 4 数据与存储设计

### 4.1 双层持久化策略

ZhiPath 采用"关系数据库 + JSON 文件"双层持久化，二者职责互补：

- **PostgreSQL（强一致、可查询、带向量索引）**承载需要跨会话共享、需要事务、需要向量检索的数据：用户账户、用户级 API 配置、会话、消息、长期记忆摘要、画像 JSON、知识库文档及其分块向量。
- **JSON 文件（会话级、高频读写、无需跨表 join）**承载每个 session 独立的状态快照：BKT 掌握度、DKT 模型状态、FSRS 复习卡片、知识图谱、资源包、测验记录、实验分桶等。每个 session 一个文件，路径用 session_id 经正则安全化（`[^a-zA-Z0-9_-]` 清除）后命名。

这一策略的好处：会话级状态读写无锁竞争；JSON 可直接 `cat` 调试；数据库压力小；pgvector 不可用时系统可降级到纯 JSON 模式继续运行。

### 4.2 数据库表（8 张，SQLAlchemy ORM）

全部 ORM 模型定义在 `backend/services/models.py`，主键统一为 UUID 字符串。

| 表名 | 主键 | 主要字段 | 说明 |
|---|---|---|---|
| `users` | id (UUID) | username(unique)、hashed_password、created_at | 用户账户 |
| `user_api_configs` | id (UUID) | user_id(FK users, CASCADE, unique)、configs_json(JSONB)、tts_json(JSONB)、updated_at | 用户级 LLM/TTS 凭据 |
| `sessions` | id (UUID) | user_id(FK users, SET NULL, nullable)、title(默认"新对话")、created_at、updated_at | 对话会话 |
| `messages` | id (UUID) | session_id(FK sessions, CASCADE)、role、content(Text)、timestamp | 消息记录 |
| `memory` | session_id (PK, FK sessions CASCADE) | summary(Text)、profile_md(Text) | 会话级记忆摘要 |
| `profiles` | session_id (PK, FK sessions CASCADE) | data(JSONB)、updated_at | 学习者画像 |
| `documents` | id (UUID) | title、content(Text)、tags(JSONB)、source(默认"manual")、created_at | 知识库文档 |
| `document_chunks` | id (UUID) | document_id(FK documents, CASCADE)、chunk_index(int)、content(Text)、embedding(Vector(768)) | 文档分块与向量 |

关系约束要点：

- `users → user_api_configs` 一对一，`ondelete=CASCADE`。
- `users → sessions` 一对多，`ondelete=SET NULL`（用户删除后会话保留）。
- `sessions → messages / memory / profiles` 级联删除（`ondelete=CASCADE`）。
- `documents → document_chunks` 级联删除。
- `document_chunks.embedding` 为 `Vector(768)`，由 pgvector 提供；测试环境无 pgvector 时回退为 `LargeBinary`。

数据库初始化在 `api/main.py` 的 `lifespan` 中调用 `services.database.init_db()`（`Base.metadata.create_all`），若驱动不可用则警告并回退 JSON 存储。

### 4.3 JSON 文件存储（18 个目录）

`backend/data/` 下共 18 个子目录，每目录对应一类会话级或全局级状态：

| 目录 | 存储内容 | 对应服务 |

## 5 核心算法详细设计

### 5.1 BKT（贝叶斯知识追踪）

实现于 `services/mastery/bkt.py`，基于 Corbett & Anderson (1995)。每个知识组件（KC）维护 4 个参数：

| 参数 | 含义 | 默认值 |
|---|---|---|
| `p_init` (P(L0)) | 接触前已掌握的先验概率 | 0.3 |
| `p_learn` (P(T)) | 单次练习后未掌握→已掌握的跃迁概率 | 0.15 |
| `p_slip` (P(S)) | 已掌握但答错的失误概率（上限 0.3） | 0.1 |
| `p_guess` (P(G)) | 未掌握但猜对的概率（上限 0.5） | 0.2 |

更新流程（`BKTTracker.update`）：

1. 取先验：首次答题用 `p_init`，否则用当前 `mastery`。
2. 贝叶斯后验：`P(L_t|obs) = P(obs|L)·P(L) / P(obs)`，分别对 correct/wrong 用失误/猜测代入。
3. 跃迁：`new_mastery = posterior + (1 - posterior) · p_learn`。
4. 夹紧到 [0,1]，记录 history（时间戳、正误、更新后 mastery）。

`MasteryStore` 提供 `get_mastery`（按 mastery 升序返回 KC 列表 + 汇总：count/avg_mastery/weak(<0.5)/mature(≥0.85)）、`upsert_kcs`、`update_observations`、`get_focus_kcs`（默认阈值 0.6，返回最薄弱的前 5 个 KC）。BKT 在 ZhiPath 中的作用：每次 Quiz 提交或 Auto-Tutor 自评都喂一组观测更新掌握度；前端"掌握度热力图"按 KC × 时间渲染学习曲线；资源生成优先针对薄弱 KC 出题。

### 5.2 DKT（深度知识追踪）

实现于 `services/mastery/dkt.py`，参考 Piech et al. (2015)，采用轻量 `MiniGRU` + numpy 实现（无 PyTorch/TF 依赖）。

- **输入**：`2 × num_kc` 维（KC one-hot 拼 correctness one-hot）。
- **隐藏**：GRU 单元，默认 `hidden=12`，Xavier 初始化。
- **输出**：`num_kc` 维 sigmoid，表示下一题各 KC 答对概率。
- **训练**：`_train_es` 用 Evolution Strategies 风格更新（随机扰动 + 选 loss 更低的），默认 `iters=40`，避免手写 BPTT。损失为逐步交叉熵。
- **推理**：`predict_after(seq)` 跑完整段答题序列后给出下一题各 KC 的 P(correct)。

`DKTService` 对每个 session 维护独立模型状态（kc_index、loss、epoch、predictions、seq_len），落盘到 `data/dkt/<session>.json`。数据量过小时自动退化为 BKT 先验。DKT 的 `predict_next` 可作为 IRT ability 估计的输入。

### 5.3 IRT（项目反应理论）

实现于 `services/mastery/irt.py`，采用 **2PL 模型**（两参数 logistic，不含猜测参数 c）：

```
P(答对 | θ, item) = 1 / (1 + exp(-a · (θ - b)))
```

- `θ`：学生能力（潜在变量）。
- `a`：题目区分度（默认 1.0），越大该题对能力越敏感。
- `b`：题目难度（默认 0.0）。

核心函数：

- `prob_correct(theta, item)`：答对概率。
- `item_information(theta, item) = a² · P · (1 - P)`：题目信息量，在 θ ≈ b（P=0.5）处最大。
- `estimate_ability(responses, prior=0, iters=30, lr=0.4)`：牛顿法 MLE 估计 θ，夹紧到 [-4, 4]。
- `select_next_item(theta, candidates)`：CAT 策略，选当前能力下信息量最大的题。
- `recommend_difficulty(theta)`：按 θ 分档返回 very_easy/easy/medium/hard/very_hard（分界点 -1.5/-0.3/0.6/1.5），供资源生成 prompt 使用。
- `mastery_to_theta(mastery)`：把 BKT mastery(0-1) 线性映射到 θ(-2.5 ~ 2.5)，`(mastery - 0.5) × 5`。

IRT 在资源生成流程中实时生效：`ResourceGenerationCapability` 读取 BKT 平均 mastery → 转 θ → 给出难度建议 → 注入 learning_doc 顶部，7 个生成器据此产出对应难度的题目。

### 5.4 FSRS（间隔重复调度）

实现于 `services/srs/fsrs.py`，基于 FSRS-4 论文的 17 参数默认值：

```
DEFAULT_PARAMS = (0.40, 0.60, 2.40, 5.80, 4.93, 0.94, 0.86, 0.01,
                  1.49, 0.14, 0.94, 2.18, 0.05, 0.34, 1.26, 0.29, 2.61)
REQUEST_RETENTION = 0.9   # 目标可提取性 90%
MAXIMUM_INTERVAL = 365 × 5  # 上限 5 年
```

三隐变量建模记忆：

- **Stability (S)**：记忆稳定性，越大遗忘越慢。
- **Difficulty (D)**：卡片难度，夹紧 [1,10]，向 `w[4]` 回归。
- **Retrievability (R)**：当前可提取性，遗忘曲线 `R = (1 + elapsed_days/(9·S))^(-1)`。

评级 `Rating` 四档（AGAIN=1/HARD=2/GOOD=3/EASY=4，与 Anki 兼容）。`FSRSScheduler.review(card, rating)` 更新流程：

1. 计算 elapsed_days（距上次复习的天数）。
2. 新卡：`_init_difficulty` 与 `_init_stability`（直接取 `w[rating-1]`）。
3. 复习卡：算 retrievability → `_next_difficulty`（向均值回归）→ AGAIN 走 `_next_forget_stability`（遗忘后新稳定性），其余走 `_next_recall_stability`（含 hard_penalty `w[15]` 与 easy_bonus `w[16]`）。
4. 计算下次复习间隔：

```
interval = stability × 9 × (1 / REQUEST_RETENTION - 1)
```

即 `stability × 9 × (1/0.9 - 1) = stability × 1.0`（保留 90% 记忆的稳态间隔），夹紧到 [1, MAXIMUM_INTERVAL]。

FSRS 在 ZhiPath 中的作用：Quiz 错题、被标记的闪卡自动入复习队列；前端"复习日历"按 due_date 把卡片摊到时间轴；学习路径提示"今天该复习 X 张卡片"。

### 5.5 RAG 管线

RAG 管线由五个组件串联，定义在 `services/rag/` 下：

1. **Embeddings**（`embeddings.py`）：`HuggingFaceEmbeddings` 加载 `sentence-transformers/all-mpnet-base-v2`，CPU 推理，`normalize_embeddings=True`，输出 768 维向量。单例缓存。
2. **Pipeline**（`pipeline.py`，约 1246 行）：核心检索与切片。
   - 切片：`_chunk_content(content, chunk_size=1000, overlap=150)`，按句子边界软切，min_size=800、max_size=1200。
   - 向量检索：pgvector 余弦相似度 top-k，`num_retrieval_results=5`。
   - Reranker 触发：`overfetch_factor=3`，先取 `k × 3` 候选再精排到 k。
   - 种子文档：内置 408 数据结构/操作系统/计算机组成/计算机网络等知识点文档。
3. **SmartRetriever**（`smart_retriever.py`）：在 GraphRAG 之前叠加"多查询变体"。用便宜 LLM 生成 2 条 query 变体（含原 query 共 3 条），各自喂 GraphRAG，合并去重按分数排序。解决"同义词/不同表述"漏召回。
4. **Reranker**（`reranker.py`）：三种策略——`crossencoder`（默认，`cross-encoder/ms-marco-MiniLM-L-6-v2`，`max_length=512`）、`llm`（用 LLM 打分）、`none`（跳过）。`top_k=5`。
5. **GraphRAG**（`graph_rag.py`）：向量召回 top-k → 把命中 chunks 映射到 KG 节点 → 1-hop 邻居扩展（前置+后继）→ 按"语义分 + 图距离衰减（hop_decay=0.5）"重排。解决"知识依赖"漏召回。

引用标注由 `services/guardrail/citation.py` 的 `build_cited_context` 完成，给检索片段加编号并标记 `low_confidence`。

### 5.6 知识图谱

实现于 `services/knowledge_graph/graph.py`。数据结构：`{nodes: [...], edges: [...], updated_at}`。

- **节点**：`{id, label, ...}`，id 由 label slugify 生成。
- **边**：`{source, target, relation}`，默认 relation 为 `prerequisite`（前驱依赖）。

核心方法：

- `upsert_nodes`：按 id 去重合并节点。
- `add_edges`：校验两端节点存在（不存在则自动补节点），按 (source,target) 去重；合并后用 `_is_dag`（拓扑排序长度等于节点数）检测环，有环则丢弃新边并告警。
- `suggest_next`：基于掌握度与前置依赖，推荐下一步该学的节点。
- `diagnose_gaps`：给定目标节点，找出缺失的前置链路（`missing_prerequisites`）。
- `topo_sort`：拓扑排序返回学习顺序。

知识图谱由 `KGGenerator` Agent（资源生成流程之一）从学习材料中抽取，也可通过 REST API 手动维护。前端 `KnowledgeGraphView` 渲染节点-边可视化。

### 5.7 画像抽取

实现于 `services/profile/service.py` 的 `LearningProfileService`（约 720 行）。采用**规则抽取 + 证据链**模式（非纯 LLM），每轮用户消息增量更新画像：

抽取的维度（字段）包括：

- `learning_goal`：学习目标（正则 + 关键词抽取）。
- `level`：能力层级（beginner/intermediate/advanced）。
- `topics`：涉及学科/主题（最多 12 个，带动态词表）。
- `weak_points`：自述薄弱点（最多 10 个）。
- `preferences`：学习偏好（最多 10 个）。
- `constraints`：时间/节奏约束（最多 8 个）。
- `recent_intents`：近期意图（最多 8 个）。
- `exam_context`：408 考研场景（exam_code、target_school、exam_stage、weak_subjects 累积去重最多 4 个、exam_date）。
- `quiz_accuracy`、`last_quiz_time`：由测验反馈回填。

**证据链机制**：每次新增维度值都写入 `evidence_log`（dimension、value、turn、snippet 原话前 160 字、capability），并建 `evidence_index`（value → 证据列表）。前端"画像证据链"面板据此展示"这个结论来自第几轮的哪句话"。增量通过 WS 的 `profile_update` 事件实时下发。

画像还支持卡片化输出：`make_profile_summary` 根据主信号/目标/冷启动三种状态生成一句话摘要；冷启动时返回 `cold_start` 卡片引导用户补全信息。

### 5.8 置信度与引用

置信度体现在三个层面：

1. **检索置信度**：Reranker 分数 + GraphRAG 图距离衰减共同决定片段排序；低于阈值的片段标记 `low_confidence=True`，WS 下发 `sources` 事件时一并告知前端，前端以淡色/警示样式渲染。
2. **画像置信度**：证据链的条数与一致性隐含置信度——同一 value 多轮多次出现则置信高；仅一轮单次出现则低。前端 `ProfileEvidencePanel` 展示证据数量。
3. **掌握度置信度**：BKT 的 `attempts` 与 `accuracy` 字段、DKT 的 `loss` 与 `seq_len` 共同反映掌握度估计的可靠度；attempts 越少估计越不确定，资源生成会相应给出更保守的难度。

引用标注：`citation.py` 给检索片段编号（如 [1]、[2]），LLM 回答时引用编号，前端 `MessageBubble` 可点击编号跳转到来源文档。

### 5.9 测验反馈闭环

实现于 `services/quiz/feedback_service.py` 的 `QuizFeedbackService.evaluate_and_update`，是连接"测验—掌握度—复习—画像"的核心闭环：

1. **答题判定**：`_flatten_questions` 展开题型，`_check_answer` 逐题判正误。
2. **BKT 更新**：每道题作为一个观测（label = 题目摘要前 30 字），调 `MasteryStore.update_observations` 更新掌握度。
3. **DKT 更新**：同一组观测调 `DKTService.fit_and_predict`，训练 MiniGRU 并给出下一题各 KC 答对概率。
4. **FSRS 入队**：`extract_review_candidates_from_quiz` 从错题提取复习卡片，调 `ReviewStore.add_cards(source="quiz")` 入复习队列。
5. **画像回填**：调 `profile_service.update_weak_points_from_quiz` 把错题主题写入 `weak_points`，更新 `quiz_accuracy` 与 `last_quiz_time`。
6. **反馈生成**：`_build_analysis` 给出准确率与错误模式分析；`_build_remediation_plan` 按准确率分档（≥0.8 掌握良好 / 0.5-0.8 部分掌握 / <0.5 需要补救）给出补救建议；`_infer_error_patterns` 推断常见错误模式。

闭环效果：一次测验同时更新了掌握度（BKT/DKT）、复习计划（FSRS）、画像（weak_points），下一轮资源生成会针对更新后的薄弱点出题，形成"测—评—练—复"的正循环。

## 6 功能模块详细设计

### 6.1 资源生成 Agent

资源生成能力（`capabilities/resource_gen.py`）通过 `asyncio.gather` 并行调度 7 个生成 Agent，每个 Agent 是 `modules/resource_gen/agents/` 下的独立模块：

| Agent | 模块文件 | 产出 |
|---|---|---|
| QuizGenerator | quiz_generator.py | 分层测验题（单选/多选/判断/简答，数量由 `_infer_quiz_counts` 推断） |
| FlashcardGenerator | flashcard_generator.py | 问答闪卡（适合 FSRS 间隔复习） |
| MindMapGenerator | mindmap_generator.py | 层级思维导图（Markdown 缩进树） |
| CodeLabGenerator | code_lab_generator.py | C 代码实操片段（可进沙箱运行） |
| MermaidGenerator | mermaid_generator.py | Mermaid 语法图表（流程图/时序图等） |
| KGGenerator | kg_generator.py | 知识图谱节点与依赖边 |
| CaseStudyGenerator | case_study_generator.py | 案例分析（真实场景应用） |

此外 `modules/resource_gen/agents/explainer.py` 提供动画讲解脚本生成，由 `ExplainerCapability` 单独调用（不在并行 gather 中）。

每个 Agent 接收统一参数：`llm` 实例、`learning_document`（学习文档）、`learner_profile`（画像）。QuizGenerator 额外接收四种题型的数量；CodeLabGenerator 额外接收 `user_request`。

### 6.2 资源生成流水线

`ResourceGenerationCapability.run` 的完整流程：

1. **组装学习文档**：若 `context.knowledge_context` 非空，则把"本轮用户需求 + 知识库参考"拼接为 learning_doc；否则用 user_message。
2. **注入 408 考研上下文**：若画像含 `exam_context.exam_code`，在 learning_doc 顶部注入考试代码、目标层次、复习阶段、弱项学科、考试日期与产出口径。
3. **IRT 自适应难度**：读取 BKT 平均 mastery → `mastery_to_theta` 转 θ → `recommend_difficulty` 得难度档位，以 `thinking` 事件下发并注入 learning_doc。
4. **并行生成**：在 `stream.stage("resource_generation")` 内，先逐个 emit `agent_message`（Orchestrator → 各 Agent，携带需求摘要），再用 `asyncio.gather` 并行执行 7 个 Agent。
5. **结果汇总**：把 7 个产出写入 `agent_results`（测验题目、复习闪卡、知识结构图、代码实操、结构化图表、案例分析、知识图谱）。若检测到试卷导出意图，额外生成"可打印试卷"结构。
6. **可选 TTS**：若讯飞 TTS 可用，为讲义合成音频。
7. **资源包持久化**：调 `ResourcePackageStore` 保存完整资源包。
8. **LLM 总结**：用 long_form 路由的 LLM 基于 agent_results 生成微讲义与个性化说明，流式输出。

### 6.3 Auto-Tutor 一键闭环

`AutoTutorCapability`（`capabilities/auto_tutor.py`）把"目标诊断 → 资源生成 → 试卷封装 → 学习建议报告"串成一条管线，用一条用户消息触发整套多智能体协作。四步定义在 `LOOP_STEPS`：

| step | 名称 | 涉及 Agent/服务 |
|---|---|---|
| `diagnose` | 目标诊断 | GoalPlanner + SkillMapper + GapAnalyzer |
| `generate` | 资源生成 | QuizGenerator + FlashcardGenerator + MindMapGenerator + CodeLabGenerator |
| `exam` | 试卷封装 | ExamStore |
| `report` | 学习建议报告 | LLM（reasoning 路由）基于前三步结构化产出撰写 |

每步通过 `bus.loop_step(step, status)` 下发 `LOOP_STEP` 事件，前端显示进度条。任何单步失败都 try/except 优雅降级、继续推进，绝不卡死管线。最后 LLM 基于已注入的各阶段结果组织"学习建议报告"（目标与差距分析、资源概览、学习建议、下一步），流式输出。

### 6.4 智能辅导（ChatCapability）

`ChatCapability`（`capabilities/chat.py`）是基础流式问答能力，name=`chat`，stage=`tutor_response`，走 `ModelRouter:chat` 路由。职责：知识讲解、追问引导、学习建议。它是所有未知能力的兜底——编排器找不到目标能力时回退到 chat。

### 6.5 Agentic 自主决策（AgenticChatCapability）

`AgenticChatCapability`（`capabilities/agentic_chat.py`）实现 LLM 自主工具调用，stage 为 `thinking / acting / responding`。

- **工具集**（8 个）：`route_to_goal`、`route_to_learning`、`route_to_resource_gen`、`route_to_auto_tutor`、`route_to_debate`（5 个能力路由）、`query_mastery`、`query_due_cards`、`query_knowledge_graph`（3 个学情查询）。
- **工具循环**：`MAX_TOOL_ITERATIONS = 4` 轮上限，每轮 `MAX_PARALLEL_TOOLS = 4` 个工具并行。LLM 若返回 tool_calls 则执行工具、回填结果、继续循环；若无 tool_calls 则流式输出最终文本。
- **降级**：LLM provider 不支持 `bind_tools` 时走 ReAct 兜底（文本解析工具调用）。
- 每次工具调用都 emit `agent_message`（AgenticChat → 工具名），前端工作流图实时高亮。

### 6.6 辩论（DebateCapability）

`DebateCapability`（`capabilities/debate.py`）实现苏格拉底式多智能体辩论，三角色 2 轮：

- **Proposer（正方）**：reasoning 路由，提出观点与论据。
- **Skeptic（反方）**：reasoning 路由，反驳正方。
- **Judge（裁判）**：long_form 路由，综合双方论据给出最终学习建议，流式输出。

流程：Round 1（正方→反方反驳）→ Round 2（正方更新→反方更新）→ Judge 终审（流式）。每步通过 `agent_message` 标注 from/to/label，transcript 完整记录每轮发言。manifest 描述："多智能体辩论：正方/反方/裁判三角色，对学习问题进行 2 轮辩论后给出结论。"

### 6.7 动画讲解（ExplainerCapability）

`ExplainerCapability`（`capabilities/explainer.py`）生成动画讲解脚本并合成语音。tools_used 为 `ExplainerAgent` 与 `iFlytekTTS`。流程：调用 `modules/resource_gen/agents/explainer.py` 生成结构化讲解脚本（含分段 narration 与 duration_ms），再调 `_synthesize_full_audio` 把全部段旁白拼接成一次 TTS 合成（返回 mp3 URL）。前端 `ExplainerPlayer` + `DigitalHumanStage3D` 按音频时长驱动数字人嘴型与场景动画。TTS 不可用时降级为纯文本讲解。

### 6.8 学习路径（LearningCapability）

`LearningCapability`（`capabilities/learning.py`）实现个性化学习路径规划，stage=`learning_plan`，tools_used=`ProfileBuilder / PathScheduler / FeedbackPlanner`。流程：

1. **ProfileBuilder**：读取/构建学习者画像（cognitive_status）。
2. **PathScheduler**：调 `modules/learning_path/agents/path_scheduler.py` 的 `schedule_learning_path_with_llm`，基于画像、目标、掌握度生成学习路径（知识点序列 + 建议时长 + 练习节奏）。
3. **FeedbackPlanner**：给出反馈策略（何时测验、何时复习）。
4. 路径结果写入 `agent_results["学习路径规划"]`，并通过 `PathRevisionStore` 保存版本历史，支持前后对比。

### 6.9 目标诊断（GoalCapability）

`GoalCapability`（`capabilities/goal.py`）实现三阶段目标诊断，stage=`goal_diagnosis`，tools_used=`GoalPlanner / SkillMapper / GapAnalyzer`：

1. **GoalPlanner**：精炼学生的学习目标（从模糊表述提取可执行目标）。
2. **SkillMapper**：把目标映射到所需技能/知识组件清单。
3. **GapAnalyzer**：对照学生当前掌握度，识别技能差距（已有 vs 应有）。

三步顺序执行（非并行），每步 emit `tool_call`/`tool_result`。产出用于驱动 Auto-Tutor 的后续资源生成与路径规划。

## 7 关键流程时序设计

### 7.1 资源生成主流程

以用户输入"帮我生成数据结构链表的学习资源"（capability=resource_gen）为例：

1. **WS 接入**：客户端发 `{type:"message", content:"...", capability:"resource_gen", session_id:"..."}`。
2. **上下文组装**（`_handle_turn`）：落库用户消息 → 安全检查 → 画像更新（下发 profile_update） → SmartRetriever 检索"链表"相关知识（下发 sources） → 组装 UnifiedContext。
3. **编排器分发**：ChatOrchestrator 按 active_capability 找到 ResourceGenerationCapability，创建 StreamBus，下发 `session` 与 `trace_start` 事件。
4. **难度估计**：能力读取 BKT mastery → IRT 推荐 difficulty → 下发 `thinking`。
5. **并行生成**：进入 `stage_start: resource_generation`，依次 emit 7 条 `agent_message`（Orchestrator → 各 Agent），随后 `asyncio.gather` 并行执行；每个 Agent 内部 emit `tool_call`/`tool_result`。
6. **结果汇总**：7 个产出写入 agent_results，emit `result` 事件。
7. **资源包持久化**：ResourcePackageStore 保存。
8. **LLM 总结**：long_form 路由 LLM 流式输出微讲义，逐块 emit `content`。
9. **结束**：`stage_end` → `done`（携带 trace_id）；WS 层把累积的 content 落库为 assistant 消息。

### 7.2 测验评估闭环

以学生提交一份 Quiz 为例（REST `POST /api/v1/quiz/.../submit`）：

1. **接收作答**：QuizStore 读取测验题目与用户答案。
2. **逐题判定**：`_check_answer` 判正误，生成 observations 列表。
3. **BKT 更新**：`MasteryStore.update_observations` 更新各 KC 掌握度，返回 mastery_snapshot（含 summary）。
4. **DKT 更新**：`DKTService.fit_and_predict` 训练 MiniGRU，返回各 KC 下一题答对概率与 loss。
5. **FSRS 入队**：错题经 `extract_review_candidates_from_quiz` 转 FSRSCard，`ReviewStore.add_cards(source="quiz")` 入复习队列并计算 due_date。
6. **画像回填**：`profile_service.update_weak_points_from_quiz` 写入 weak_points、quiz_accuracy。
7. **反馈生成**：`_build_analysis` + `_build_remediation_plan` + `_infer_error_patterns` 生成结构化反馈（准确率、错误模式、补救计划、掌握档位）。
8. **xAPI 记录**：（可选）LRSStore.emit verb="answered"，记录本次测验的 result（score、success、duration）。

闭环完成：掌握度、复习计划、画像、学习记录四处状态同步更新，下一轮资源生成据此个性化。

### 7.3 Auto-Tutor 一键闭环

以用户输入"我想 3 个月搞定 408 数据结构"（capability=auto_tutor）为例：

1. **WS 接入 + 上下文组装**：同 7.1 第 1-2 步。
2. **LOOP_STEP: diagnose**：`bus.loop_step("diagnose", "running")`，执行 GoalPlanner → SkillMapper → GapAnalyzer 三步，产出目标、技能清单、差距分析；`loop_step("diagnose", "done")`。
3. **LOOP_STEP: generate**：并行执行 QuizGenerator + FlashcardGenerator + MindMapGenerator + CodeLabGenerator，产出题目、闪卡、导图、代码片段；单步失败降级继续。
4. **LOOP_STEP: exam**：把生成的题目封装成试卷（ExamStore），含题量与满分。
5. **LOOP_STEP: report**：把前三步结构化结果注入 system_prompt，reasoning 路由 LLM 流式输出"学习建议报告"（目标与差距分析、资源概览、学习建议、下一步），逐块 emit `content`。
6. **结束**：`done`（携带 trace_id）。

整个闭环前端通过 `loop_step` 事件渲染四步进度条，通过 `agent_message` 渲染各阶段 Agent 协作图。

## 9 非功能设计实现

### 9.1 流式与进度追踪

- **流式输出**：所有能力通过 StreamBus emit `content` 事件逐块下发，WS 层 `_stream_orchestrator` 累积 chunks 拼成完整回复。LLM 调用统一用 LangChain 的 `astream` 接口。
- **阶段进度**：`StreamBus.stage(name)` 上下文管理器自动 emit `stage_start`/`stage_end`，前端据此渲染阶段边界。
- **闭环进度**：Auto-Tutor 用 `loop_step` 事件下发四步进度（running/done）。
- **全链路 trace**：每轮生成 `turn_id`（UUID），`trace_scope(turn_id)` + `tracing_span(name, kind, attributes)` 把所有事件打上 trace_id，前端 `TraceTimelineView` 与 `LiveTelemetryHUD` 实时展示。

### 9.2 防幻觉

防幻觉通过四道防线：

1. **RAG 接地**：回答前先检索知识库，把检索片段作为 context 注入 prompt，约束 LLM 基于给定材料回答。
2. **引用标注**：`citation.py` 给检索片段编号，要求 LLM 在回答中引用编号，前端可溯源。
3. **低置信标记**：检索分数低的片段标记 `low_confidence`，前端以警示样式呈现，提示学生核实。
4. **温度置零**：所有 LLMProfileConfig 默认 `temperature=0`，降低随机性。

### 9.3 内容安全

实现于 `services/guardrail/safety.py`，本地正则级双模式检测，覆盖输入与输出：

- **block 模式（7 条，命中即拒绝）**：涉政暴恐（颠覆国家政权、煽动暴力、自杀指南、制造炸药、未成年色情）+ 提示词注入（`ignore.*above.*instructions`、`jailbreak mode`）。
- **warn 模式（4 条，放行但提示）**：学术不端（抄袭、代写、考试作弊、找枪手）。

WS 层 `_check_safety` 在落库用户消息后立即检查，命中 block 则下发 `guardrail` 事件（severity/reason/matched）+ 友好拒绝文案 + `done`，终止本轮。扩展点：可接入讯飞内容审核 API（`_remote_check`）。

### 9.4 多模型智能路由

实现于 `base/model_router.py`。按任务类型路由到不同 LLM profile，每个路由有 primary 与 fallbacks：

| 任务 | primary | fallbacks | 说明 |
|---|---|---|---|
| chat | deepseek-chat | qwen-turbo、qwen2.5-7b、iflytek-spark-lite | 日常导学对话，快且便宜 |
| structured | deepseek-chat | qwen-plus、deepseek-v3-sf、glm-4-9b | 结构化 JSON（quiz/flashcard/mermaid） |
| long_form | deepseek-chat | qwen-max、qwen-plus、deepseek-v3-sf | 长文/讲义/报告 |
| reasoning | deepseek-reasoner | deepseek-chat、qwen-max、deepseek-v3-sf | 目标诊断/差距分析/闭环报告 |
| code | deepseek-chat | qwen-plus、deepseek-v3-sf、qwen2.5-7b | CodeLab 代码合成 |
| mermaid | deepseek-chat | qwen-plus、iflytek-spark-pro、glm-4-9b | Mermaid 图表（语法严格） |

路由优先级：① 用户在前端配置的 ApiConfig（按 task 匹配）→ ② config/default.yaml 的 profile → ③ fallbacks 依次尝试。`GET /api/v1/router` 暴露路由表与最近 50 条路由日志，便于评审观察模型选择策略。

### 9.5 凭据健康监测

ChatOrchestrator 在能力执行异常时检测错误关键词并 emit `CREDENTIAL_HEALTH` 事件：

| 错误特征 | content | error_code |
|---|---|---|
| 429 / rate limit / too many requests | `rate_limited` | 429 |
| 401 / 403 / unauthorized / forbidden / invalid api key / authentication | `auth_error` | 401 |
| connection / timeout / refused / network | `connection_error` | 0 |

前端收到后弹出凭据健康提示（如"当前模型限流，建议切换"），引导用户在 `CredentialSettingsPanel` 切换配置。

### 9.6 全链路追踪

实现于 `services/tracing/tracer.py`。提供 `trace_scope(trace_id)` 上下文与 `span(name, kind, attributes)` 上下文。每轮对话生成唯一 turn_id，所有事件（StreamEvent）都打上 session_id 与 turn_id。WS 下发 `trace_start`（携带 trace_id）与 `done`（携带 trace_id）作为边界。前端 `TraceTimelineView` 按 trace_id 渲染该轮所有 span 的时间线，`LiveTelemetryHUD` 实时展示当前进行中的 span。REST `GET /api/v1/tracing/...` 可按 trace_id 查询历史。

### 9.7 降级容错

系统在多处实现优雅降级：

- **数据库不可用**：`init_db` 失败则警告并回退 JSON 存储；`Vector` 导入失败回退 `LargeBinary`。
- **LLM 凭据缺失**：能力内 try/except，失败 emit `error` 但保证 `done`；编排器兜底 emit DONE。
- **Reranker 未安装**：`get_reranker` 返回 `NoneReranker`，跳过精排直接返回 top-k。
- **TTS 不可用**：`iflytek_tts_available()` 为假时跳过音频合成，降级纯文本。
- **Auto-Tutor 单步失败**：每步独立 try/except，失败继续推进下一步。
- **未知能力**：编排器回退到 `chat` 能力。
- **bind_tools 不支持**：Agentic 走 ReAct 文本解析兜底。

### 9.8 代码沙箱隔离

学生 C 代码运行（`services/code_lab/`）采用双重防线：

**第一道：静态护栏**（`guardrail/code_safety.py`）

- 头文件白名单（12 个标准库头）：stdio.h、stdlib.h、string.h、math.h、stdbool.h、stddef.h、stdint.h、inttypes.h、limits.h、ctype.h、time.h、assert.h。
- 危险函数黑名单（按 `\bname\s*\(` 匹配）：system/popen/exec*/fork/vfork/spawn*、fopen/open/creat/close/remove/unlink/rename/mkdir/rmdir/chdir、Windows API（CreateFile/CreateProcess/ShellExecute/LoadLibrary 等）、网络（socket/connect/bind/recv/send）、getenv、signal/raise/abort/exit、setjmp/longjmp、alloca、内联汇编 asm。
- 禁止 `#define` 宏与 `##` token 粘贴（防宏绕过）。
- 剥离注释与字符串字面量后再匹配，避免藏在字符串里逃逸。

**第二道：运行时隔离**（`code_lab/runner.py`）

- 编译器自动探测（缓存）：按 `compiler_preference=("tcc","gcc","clang","cl")` 顺序在 PATH 探测；可用 `compiler_path` 或环境变量 `ZHIPATH_C_COMPILER` 显式指定。
- 每次运行用 `mkdtemp` 独立临时目录，finally 清理。
- 硬超时 `timeout_seconds=5.0`，输出截断 `max_output_bytes=65536`。
- Windows 下新进程组（CREATE_NEW_PROCESS_GROUP），超时按树状 kill。
- 编码 errors=replace，避免乱码崩溃。
- RunResult 含 reason 分类：ok / runtime_error / compile_error / timeout / no_compiler / blocked。

## 10 部署与运行设计

### 10.1 组件与端口

Docker Compose 编排三个服务（`docker-compose.yml`）：

| 服务 | 镜像/构建 | 端口 | 依赖 |
|---|---|---|---|

| `backend` | ./backend/Dockerfile | 8000:8000 | postgres（healthy） |
| `frontend` | ./frontend/Dockerfile | 3000:3000 | backend（healthy） |

环境变量：

- postgres：`POSTGRES_USER=zhipath`、`POSTGRES_PASSWORD=zhipath`、`POSTGRES_DB=zhipath`。
- backend：`DATABASE_URL=postgresql+asyncpg://zhipath:zhipath@postgres:5432/zhipath`；可选凭据 `DEEPSEEK_API_KEY`、`DASHSCOPE_API_KEY`、`SILICONFLOW_API_KEY`、`XF_SPARK_API_PASSWORD`、`XF_TTS_APPID/API_KEY/API_SECRET`（未设置则相关功能优雅降级）。

### 10.2 配置驱动

配置分三层：

1. **config/default.yaml**：LLM profiles（DeepSeek/通义千问/硅基流动/讯飞星火等 provider 与 model_name、base_url、api_key_env）、embedding、reranker、search、rag、code_lab 等默认值。
2. **config/schemas.py**：`AppConfig` dataclass，含 `LLMConfig`（default_profile + profiles dict）、`EmbeddingConfig`（provider="huggingface"、model_name="sentence-transformers/all-mpnet-base-v2"）、`RerankerConfig`（enabled=True、strategy="crossencoder"、model_name="cross-encoder/ms-marco-MiniLM-L-6-v2"、overfetch_factor=3、max_length=512）、`SearchConfig`（provider="duckduckgo"、max_results=5）、`RAGConfig`（chunk_size=1000、num_retrieval_results=5）、`CodeLabConfig`（compiler_preference、timeout_seconds=5.0、max_output_bytes=65536）、`database_url`。
3. **用户级配置**：存于 `user_api_configs` 表（configs_json JSONB + tts_json JSONB），运行时通过 WS `init` 或 REST `X-LF-*` 头注入 contextvars，优先级高于 default.yaml。

`config/loader.py` 的 `get_config()` 单例读取 default.yaml 并构造 AppConfig。本地开发默认 `database_url="postgresql+asyncpg://postgres:123456@localhost:5433/zhipath"`（schemas.py 内置），生产/Docker 由环境变量覆盖。

### 10.3 数据库初始化

- **ORM 建表**：`api/main.py` 的 `lifespan` 启动时调 `services.database.init_db()`（`Base.metadata.create_all`），自动创建 8 张表。驱动不可用则警告并回退 JSON。
- **向量扩展**：pgvector 镜像自带扩展，`document_chunks.embedding` 列类型为 `Vector(768)`。
- **种子数据**：`RAGPipeline()` 在 lifespan 中实例化，首次启动时把内置 408 知识点文档（数据结构、操作系统、计算机组成、计算机网络等）切片、嵌入、写入 documents/document_chunks 表。`scripts/ingest_408.py`、`ingest_course.py`、`ingest_pdf.py` 提供批量导入入口；`verify_408.py` 校验导入完整性。
- **Alembic 迁移**：`backend/alembic/env.py` 提供迁移脚本入口（生产环境推荐用 Alembic 而非 create_all）。
