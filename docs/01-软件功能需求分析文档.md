## 1 引言

### 1.1 编写目的

本文档对 ZhiPath 系统的功能需求、非功能需求、数据需求及接口需求进行系统化定义，作为参赛提交、架构设计、开发实现与测试验收的共同依据。文档面向以下读者：

- **评审专家**：了解系统功能边界与技术深度，作为评审打分依据。
- **开发团队**：作为前后端模块开发、接口联调与测试用例编写的基础。
- **运维人员**：作为部署配置、环境依赖与监控埋点的参考。

### 1.2 项目背景

A3 赛题要求"基于大模型的个性化资源生成与学习"。传统在线教育平台普遍存在三大痛点：

1. **资源静态化**：题库、讲义、思维导图等内容由人工预制，无法针对个体学情动态生成。
2. **学情模糊化**：缺乏对学习者知识状态、认知偏好、学习目标的持续建模，导致推荐"千人一面"。
3. **反馈滞后化**：学习者练习后难以及时获得解释、追问与针对性补救。

ZhiPath 以大语言模型（LLM）为认知引擎，融合知识追踪（BKT/DKT）、项目反应理论（IRT）、间隔重复算法（FSRS-4.5）与知识图谱，通过 9 个能力（Capability）与 7 类资源生成 Agent 构建多智能体协作系统，实现"目标诊断 → 画像建模 → 路径规划 → 资源生成 → 练习巩固 → 复习追踪"的闭环学习。

### 1.3 系统范围

系统提供以下核心能力闭环，但不包含：教务管理、收费支付、移动端原生 App、离线安装包。

| 范围类别 | 说明 |
|---------|------|
| 核心功能 | 画像构建、目标诊断、学习路径规划、资源生成、自动辅导、苏格拉底辩论、动画讲解 |
| 学习科学 | BKT/DKT 知识追踪、IRT 自适应测试、FSRS-4.5 间隔重复 |
| 资源类型 | 8 类：quiz / flashcard / mindmap / code_lab / mermaid / kg / case_study / explainer |
| 能力系统 | 9 个 Capability：chat / goal / learning / resource_gen / auto_tutor / agentic / debate / explainer / engineering_trace |
| 技术底座 | RAG 检索增强、知识图谱、Code Lab 代码沙箱、Guardrail 护栏、MCP 协议、xAPI 学习记录 |

### 1.4 术语与缩略语

| 术语 | 全称 / 说明 |
|------|------------|
| LLM | Large Language Model，大语言模型 |
| RAG | Retrieval-Augmented Generation，检索增强生成 |
| BKT | Bayesian Knowledge Tracing，贝叶斯知识追踪 |
| DKT | Deep Knowledge Tracing，深度知识追踪 |
| IRT | Item Response Theory，项目反应理论 |
| FSRS | Free Spaced Repetition Scheduler，自由间隔重复调度器 |

| xAPI | Experience API，学习体验记录规范 |

### 1.5 参考资料与开源声明

- 本项目后端基于 FastAPI、LangChain、SQLAlchemy、Alembic 等开源框架构建。
- 前端基于 Next.js 15、React 19、TypeScript、Tailwind CSS、Three.js 构建。
- 数据库使用 PostgreSQL + pgvector 扩展（`pgvector/pgvector:pg16`）。
- 嵌入模型 `sentence-transformers/all-mpnet-base-v2`，Reranker `crossencoder` 系列均来自 HuggingFace 开源生态。
- 学习科学算法（BKT/DKT/IRT/FSRS-4.5）均为公开学术成果的开源实现。
- FSRS-4.5 参考开源项目 `open-spaced-repetition`。

## 2 总体描述

### 2.1 产品定位

ZhiPath 是面向 C 语言程序设计学习场景（可扩展至其他学科）的智能学习伴侣系统。它以学习者目标为起点，以学情画像为核心，以资源生成与复习追踪为抓手，构建"感知—决策—执行—反馈"的完整学习闭环。

### 2.2 用户类与特征

| 用户类 | 典型场景 | 使用方式 |
|--------|---------|---------|
| 主要用户：学习者 | C 语言自主学习、薄弱点补救、备考刷题 | Web 浏览器访问工作台，与多智能体对话交互 |
| 次要用户：课程教师 | 观察学生学习轨迹、生成教学素材 | 通过仪表盘查看学情、导出资源 |
| 开发者/管理员 | 系统部署、能力扩展、实验配置 | 通过环境变量、API 与 JSON 配置进行运维 |

### 2.3 运行环境

#### 2.3.1 硬件环境

- CPU 4 核以上，内存 8 GB 以上，磁盘 40 GB 以上。
- 支持 GPU（可选，用于本地嵌入模型加速）。

#### 2.3.2 软件环境

| 层 | 技术栈 |
|----|--------|
| 后端 | Python 3.11 + FastAPI + LangChain + SQLAlchemy + Alembic |
| 前端 | Next.js 15 + React 19 + TypeScript + Tailwind CSS + Three.js |
| 数据库 | PostgreSQL 16 + pgvector（768 维向量） |
| 部署 | Docker Compose（三服务）；同时支持本地裸机启动 |
| 容器镜像 | postgres: `pgvector/pgvector:pg16` |

#### 2.3.3 部署端口

| 服务 | 容器端口 | 主机端口 |
|------|---------|---------|
| PostgreSQL | 5432 | 5432 |
| Backend | 8000 | 8000 |
| Frontend | 3000 | 3000 |

### 2.4 约束

- 系统通过用户自带的 API Key 调用大模型服务，支持 DeepSeek、DashScope（通义千问）、SiliconFlow、讯飞星火等供应商。
- 不内置任何需要联网下载的模型权重；嵌入模型首次启动时自动拉取。
- 当 PostgreSQL 不可用时，自动降级为 JSON 文件存储（`backend/data/` 下 18 个子目录）。
- 资源生成内容须通过 Guardrail 护栏（内容安全 7 条 block + 4 条 warn 模式、代码安全静态检查）。

### 2.5 假设与依赖

- 假设学习者具备基础计算机操作能力，并能提供有效的 API Key。
- 依赖外部 LLM 服务的可用性与稳定性；当外部服务不可用时，系统进入降级模式。
- 依赖 Docker Engine 与 Docker Compose（容器化部署方式）。

## 3 功能需求

系统功能需求按业务闭环划分为 FR-01 至 FR-13 共 13 条。每条需求包含功能描述、输入、处理逻辑、输出与验收标准。

### FR-01 学习者画像构建

#### 功能描述

系统通过对话与学习行为持续构建学习者画像，画像包含 **7 个维度**：学习目标、当前水平、关注主题、薄弱点、学习偏好、时间约束、最近意图。画像支持证据溯源，每一项结论可关联到具体对话片段或学习行为。

#### 输入

- 学习者与系统的自然语言对话内容。
- xAPI 学习行为记录（答题、复习、学习会话）。
- 显式触发：`POST /api/profile/{session_id}/extract`。

#### 处理逻辑

1. 持久化对话消息至 `messages` 表。
2. 调用画像抽取能力，按 7 维度结构化提取画像字段，写入 `profiles.data`（JSONB）。
3. 画像变更写入 `profile` 证据表，支持时间线追溯。
4. 通过 WebSocket 广播 `PROFILE_UPDATE` 事件，前端实时刷新画像雷达图。

#### 输出

- `profiles` 表（`data` 为 JSONB，含 7 维度结构化字段）。
- 画像证据 API：`GET /api/profile/{session_id}/evidence`。
- 画像上下文 API：`GET /api/profile/{session_id}/context`。

#### 验收标准

- 画像字段完整覆盖 7 个维度，且每项维度至少 1 条证据。
- 画像更新后前端在 1 秒内收到 `PROFILE_UPDATE` 事件并刷新。

### FR-02 学习目标诊断与技能差距分析

#### 功能描述

系统通过三阶段管线对学习者目标进行结构化诊断：

1. **目标精炼**（`learning_goal_refiner.py`）：将模糊目标转化为可量化、可执行的学习目标。
2. **技能需求映射**（`skill_requirement_mapper.py`）：将目标映射为所需技能点集合。
3. **差距识别**（`skill_gap_identifier.py`）：对比学员画像，识别已具备与待补齐的技能差距。

对应能力为 `GoalCapability`，内部链路为 `GoalPlanner → SkillMapper → GapAnalyzer`。

#### 输入

- 学习者自由表达的学习目标（自然语言）。
- 当前学习者画像（来自 FR-01）。

#### 处理逻辑

1. `GoalPlanner` 对原始目标进行意图识别与结构化。
2. `SkillMapper` 基于知识库检索将目标拆解为知识点/技能项。
3. `GapAnalyzer` 与画像当前水平对比，输出差距矩阵。

#### 输出

- 结构化目标 JSON、技能需求列表、差距矩阵。
- 差距结果驱动 FR-03 学习路径规划。

#### 验收标准

- 任意输入目标经管线后输出包含"目标 / 所需技能 / 已具备 / 待补齐"四类信息。
- 三阶段日志可通过工程链路追踪（FR-13）回溯。

---

### FR-03 个性化学习路径规划

#### 功能描述

基于差距分析结果，系统生成可执行的阶段性学习路径，并支持路径修订留痕。

#### 输入

- 差距矩阵（来自 FR-02）。
- 知识图谱拓扑（`GET /api/kg/{session_id}/topo_order`）。

#### 处理逻辑

1. 依据知识图谱拓扑序与学习者时间约束，编排学习阶段。
2. 每阶段绑定知识点、推荐资源类型与预期产出。
3. 路径变更写入 `path_revisions`（JSON 文件存储目录）。

#### 输出

- 路径修订列表 API：`GET /api/path/{session_id}/revisions`。

#### 验收标准

- 路径包含至少 2 个阶段，每阶段含知识点与资源建议。
- 路径调整后产生新的 revision 记录。

### FR-04 多智能体资源生成

#### 功能描述

系统以多智能体并行方式生成 **8 类学习资源**，由 `ResourceGenerationCapability`（`resource_gen`）调度。7 个生成 Agent 并行工作，`explainer` 作为第 8 类独立资源。

| 序号 | Agent 文件 | 资源类型 | 说明 |
|------|-----------|---------|------|
| 1 | quiz_generator.py | quiz | 选择题 / 测验 |
| 2 | flashcard_generator.py | flashcard | 闪卡（FSRS 复习卡片源） |
| 3 | mindmap_generator.py | mindmap | Mermaid 思维导图 |
| 4 | code_lab_generator.py | code_lab | C 编程实操（配合 CodeLab 沙箱） |
| 5 | mermaid_generator.py | mermaid | Mermaid 流程 / 时序图 |
| 6 | kg_generator.py | kg | 知识点关系图谱 |
| 7 | case_study_generator.py | case_study | 案例研究 |
| 8 | explainer.py | explainer | 动画讲解脚本（分帧 + 旁白 + TTS） |

#### 输入

- 学习者当前学习目标与薄弱点。
- RAG 检索召回的知识库片段。

#### 处理逻辑

1. `ResourceGenerationCapability` 接收生成请求，分发任务。
2. 7 个 Agent 并行执行，通过 WebSocket 推送 `STAGE_START` / `STAGE_END` / `CONTENT` 事件。
3. 生成内容经 Guardrail 护栏校验后落地至 `resource_packages` 目录。

#### 输出

- 资源包 API：`GET /api/resources`、`GET /api/resources/session/{id}/latest`、`GET /api/resources/{id}`。
- 从最近一次生成派生：`POST /api/resources/session/{id}/from-latest`。

#### 验收标准

- 一次生成请求可并行产出至少 3 类资源。
- 每类资源内容通过内容安全与（涉及代码时）代码安全检查。

### FR-05 知识追踪与掌握度建模

#### 功能描述

系统提供 **4 套学习科学算法**对学习者掌握度建模，并对外暴露统一接口。

| 算法 | 实现文件 | 核心参数 |
|------|---------|---------|
| BKT | services/mastery/bkt.py | 4 参数：P(L0) / P(T) / P(S) / P(G) |
| DKT | services/mastery/dkt.py | RNN / LSTM 序列建模 |
| IRT | services/mastery/irt.py | 3 参 logistic：难度 b / 区分度 a / 猜测 c |
| FSRS-4.5 | services/srs/fsrs.py | 17 权重；Stability / Difficulty / Retrievability；target_retention=0.9 |

#### 输入

- 答题结果（来自 FR-06 测验与 FR-08 间隔重复）。
- 学习行为 xAPI 记录。

#### 处理逻辑

1. BKT/DKT 基于答题序列更新知识点掌握概率。
2. IRT 进行能力估计与自适应选题（`POST /api/irt/{id}/select_next`）。
3. FSRS-4.5 基于记忆稳定性调度复习卡片到期时间。

#### 输出

- 掌握度 API：`GET /api/mastery/{id}`、`GET /api/mastery/{id}/focus`。
- DKT 拟合：`POST /api/dkt/{id}/fit`、`GET /api/dkt/{id}`。
- IRT 能力：`GET /api/irt/{id}/ability`。

#### 验收标准

- 答题后 BKT/DKT 掌握度更新且单调合理。
- IRT 自适应选题在 5 题内收敛到与真实能力接近的水平。
- FSRS 调度的到期卡片可通过 `GET /api/review/{id}/due` 查询。

### FR-06 测验与考试生成

#### 功能描述

系统基于画像与掌握度动态生成测验，并支持将测验组装为正式考试（可导出 docx / 打印版）。

#### 输入

- 学习者当前掌握度与薄弱点。
- 触发：`POST /api/quiz/submit`（提交答题）、`POST /api/exams/session/{id}/from-latest-quiz`（组装考试）。

#### 处理逻辑

1. `quiz_generator` Agent 生成题目。
2. 提交答案后更新掌握度（联动 FR-05）。
3. 可将最近一次测验组装为考试并导出。

#### 输出

- 最新测验：`GET /api/quiz/{session_id}/latest`。
- 考试：`GET /api/exams/session/{id}/latest`、`GET /api/exams/{id}`、`GET /api/exams/{id}/docx`、`GET /api/exams/{id}/print`。

#### 验收标准

- 测验题目难度与 IRT 能力估计匹配。
- 考试 docx 可成功导出且排版正确。

### FR-07 间隔重复复习系统

#### 功能描述

基于 FSRS-4.5 算法构建间隔重复复习系统，管理复习卡片到期调度、复习日历与评分反馈。

#### 输入

- 闪卡（来自 FR-04 flashcard 生成）。
- 测验错题导入：`POST /api/review/{id}/import_quiz`。
- 复习评分：`POST /api/review/{id}/cards/{card_id}/rate`。

#### 处理逻辑

1. FSRS-4.5 根据卡片 Stability/Difficulty/Retrievability 与 target_retention=0.9 计算下次到期。
2. 到期卡片进入复习牌组。
3. 学习者评分（Again/Hard/Good/Easy）反哺记忆模型。

#### 输出

- 到期卡片：`GET /api/review/{id}/due`。
- 复习牌组：`GET /api/review/{id}/deck`。
- 复习日历：`GET /api/review/{id}/calendar`。

#### 验收标准

- 新卡片首次评分后生成合理的下次到期时间。
- 复习日历正确反映未来 30 天的到期分布。

### FR-08 知识图谱构建与诊断

#### 功能描述

系统为每个学习会话构建独立的知识图谱（节点 + 边），支持拓扑排序、学习建议与节点级诊断。

#### 输入

- 学习内容、对话与资源生成结果。
- 通过 `POST /api/kg/{session_id}/ingest` 摄入知识。

#### 处理逻辑

1. 维护节点（`POST /api/kg/{session_id}/nodes`）与边（`POST /api/kg/{session_id}/edges`）。
2. 计算拓扑序：`GET /api/kg/{session_id}/topo_order`。
3. 基于掌握度生成学习建议：`GET /api/kg/{session_id}/suggest`。
4. 节点级诊断：`GET /api/kg/{session_id}/diagnose/{node_id}`。

#### 输出

- 图谱数据：`GET /api/kg/{session_id}`。

#### 验收标准

- 图谱节点数 ≥ 5 且存在前驱/后继关系。
- 诊断接口能针对指定节点输出掌握状态与建议。

### FR-09 知识库管理与 RAG 检索

#### 功能描述

系统提供知识库文档管理与混合检索（向量 + BM25 + 知识图谱）能力，支撑资源生成与问答。

#### 输入

- 文档上传：`POST /api/knowledge/documents`（title / content / tags / source）。
- 检索查询：`GET /api/knowledge/search`。

#### 处理逻辑

1. 文档入库后分块，嵌入模型 `all-mpnet-base-v2` 生成 768 维向量存入 `document_chunks.embedding`（pgvector）。
2. `SmartRetriever` 执行混合检索（overfetch_factor=3），Reranker 支持 `crossencoder / llm / none` 三策略。
3. GraphRAG 扩展知识图谱邻居节点增强上下文。

#### 输出

- 文档列表：`GET /api/knowledge/documents`。
- 拓扑/语义地图：`GET /api/knowledge/topology`、`GET /api/knowledge/semantic_map`。
- 查询投影与学习历史：`GET /api/knowledge/project_query`、`GET /api/knowledge/learning_history`。

#### 验收标准

- 上传文档后可在 5 秒内完成分块与嵌入。
- 检索结果 Top-3 命中率 ≥ 80%（领域内测试集）。

#### 数据实体

| 字段 | 类型 | 说明 |
|------|------|------|
| document_id | UUID (FK) | 所属文档 |
| chunk_index | int | 分块序号 |
| content | text | 分块文本 |
| embedding | Vector(768) | 嵌入向量 |

### FR-10 Code Lab 代码沙箱

#### 功能描述

系统提供 C 语言代码在线编译运行沙箱，用于 code_lab 类资源实操，配套代码安全静态检查与运行结果验证。

#### 输入

- 学习者提交的 C 源代码。
- 触发：`POST /api/resources/code-lab/run`。

#### 处理逻辑

1. `code_safety.py` 进行白名单 header 校验与危险函数拦截。
2. `runner.py` 探测可用编译器（tcc / gcc / clang / cl），编译运行（timeout=5s，max_output=64KB）。
3. `verify.py` 校验运行结果与预期的一致性。
4. `suitability.py` 评估代码适合性。

#### 输出

- 编译输出、运行结果、验证结论、安全告警。

#### 验收标准

- 正常 C 程序在 5 秒内返回正确输出。
- 含危险函数（如 `system`）的代码被拦截并给出告警。
- 输出超过 64KB 时被截断并提示。

---

### FR-11 智能体协作与工具调用

#### 功能描述

系统提供 LLM 自主工具调用能力（`AgenticChatCapability`），优先使用 native function calling，失败时回退至 ReAct 范式。同时支持 MCP 协议适配，将系统能力暴露为标准化工具。

#### 输入

- 学习者自然语言指令。
- 可用工具集（包括 MCP 暴露的 4 个工具）。

#### 处理逻辑

1. LLM 根据指令自主选择并调用工具。
2. MCP 适配层（`/api/mcp/`）暴露 4 个工具：`query_mastery` / `query_due_cards` / `query_knowledge_graph` / `recommend_difficulty`。
3. 工具调用过程通过 WebSocket 广播 `TOOL_CALL` / `TOOL_RESULT` 事件。

#### 输出

- 工具调用结果与最终回复。
- MCP 工具列表：`GET /api/mcp/`。

#### 验收标准

- 至少 80% 的可工具化指令被正确路由到工具调用。
- native function calling 不可用时自动回退 ReAct 且仍能完成任务。

---

### FR-12 学习记录与可观测性

#### 功能描述

系统对学习行为与工程链路进行全链路记录与追踪，支撑学情分析与系统可观测性。包含四项子能力：

1. **xAPI 学习记录**：`POST /api/xapi/statements`、`GET /api/xapi/{session_id}/statements`。
2. **分布式追踪**：`GET /api/tracing`、`GET /api/tracing/{trace_id}`。
3. **学习会话（番茄钟）**：`POST /api/study_session/pomodoro`、`GET /api/study_session/{id}/stats`。
4. **A/B 实验**：`GET /api/experiments`、`GET /api/experiments/{name}/results`、`POST /api/experiments/observe`。

#### 输入

- 学习行为事件（答题、复习、学习会话起止）。
- 工程链路埋点（能力调用、工具调用、阶段起止）。

#### 处理逻辑

1. xAPI 语句标准化后存入 `xapi` 目录。
2. 追踪数据按 trace_id 聚合。
3. 番茄钟记录学习时长与专注度。
4. A/B 实验按实验名归组观测指标。

#### 输出

- 学习记录、追踪详情、学习会话统计、实验结果。

#### 验收标准

- 一次完整学习闭环产生 ≥ 1 条 xAPI 语句与 ≥ 1 条 trace。
- 番茄钟结束后正确输出专注时长。

---

### FR-13 工程链路追踪展示

#### 功能描述

对应 `EngineeringTraceCapability`（`engineering_trace`），将多智能体协作的全链路（阶段起止、思考过程、工具调用、内容生成）可视化展示，供学习者与开发者理解系统行为。

#### 输入

- 各 Capability 执行过程中的事件流。

#### 处理逻辑

1. 收集 `STAGE_START` / `STAGE_END` / `THINKING` / `TOOL_CALL` 等事件。
2. 按 trace_id 聚合为时序链路。
3. 前端 `TraceTimelineView` 组件渲染。

#### 输出

- 链路时间线视图（前端 `dashboard/TraceTimelineView`）。

#### 验收标准

- 任意能力执行后可在追踪视图中看到完整阶段链路。
- 链路节点可展开查看思考与工具调用详情。

---

### FR-14 内容安全与护栏（支撑性需求）

#### 功能描述

系统通过 Guardrail 护栏对 LLM 输出进行三层约束，作为所有生成类功能的底层支撑。

| 护栏类型 | 实现文件 | 规则 |
|---------|---------|------|
| 内容安全 | services/guardrail/safety.py | 7 条 block 模式（含 prompt 注入拦截）+ 4 条 warn 模式 |
| 代码安全 | services/guardrail/code_safety.py | C 代码白名单 header + 危险函数拦截 |
| 引用标注 | services/guardrail/citation.py | 生成内容标注来源 |

#### 验收标准

- 含 prompt 注入特征的输入被 block 拦截。
- C 代码危险调用被识别并告警。
- 引用 RAG 内容的输出附带来源标注。

## 4 非功能需求

### 4.1 性能需求

| 指标 | 要求 |
|------|------|
| 流式问答首字延迟 | ≤ 2 秒（取决于 LLM 供应商） |
| 资源生成（单类） | ≤ 30 秒 |
| RAG 检索 Top-10 | ≤ 1.5 秒 |
| Code Lab 编译运行 | ≤ 5 秒（timeout 上限） |
| 前端页面首屏 | ≤ 3 秒 |

### 4.2 安全需求

- 用户密码使用 bcrypt 哈希存储（`users.hashed_password`）。
- API Key 由用户自行配置并加密存储于 `user_api_configs.configs_json`。
- 内容安全护栏拦截 prompt 注入与违规内容（7 条 block 模式）。
- 代码沙箱对危险函数与白名单外 header 进行拦截。
- Code Lab 运行设置 timeout=5s、max_output=64KB 双重限制。

### 4.3 可用性需求

- 支持 Docker Compose 一键部署与本地裸机启动两种方式。
- PostgreSQL 不可用时自动降级为 JSON 文件存储，保障核心功能可用。
- 前端 9 个页面均提供响应式布局。

### 4.4 可靠性需求

- WebSocket 断连后自动重连。
- LLM 调用失败时返回结构化错误（`ERROR` 事件）而非崩溃。
- 数据库迁移通过 Alembic 版本化管理。

### 4.5 可维护性需求

- 后端按 `capabilities / services / modules / api` 清晰分层。
- 23 个 API 路由文件按业务域拆分，统一挂载于 `api/main.py`。
- 配置项通过环境变量与 `GET /api/settings/status` 暴露健康状态。

### 4.6 开源合规

- 所有使用的开源框架（FastAPI、LangChain、Next.js、React、Tailwind、Three.js、pgvector 等）均符合其许可证要求。
- 学习科学算法均为公开学术成果的开源实现。
- 项目文档中明确标注第三方组件来源。
